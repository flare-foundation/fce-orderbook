# Orderbook Extension Design Spec

## Overview

A clean, well-documented orderbook extension for Flare Confidential Compute (FCC), built on the extension-scaffold. Implements a multi-pair limit/market order matching engine with ERC20 deposits, balance management, and TEE-authorized withdrawals.

Two instruction paths:
- **On-chain instructions** (via InstructionSender contract): deposits and withdrawals. User identity from `claimBackAddress`.
- **Direct instructions** (off-chain, encrypted): order placement, cancellation, state queries, history export. User identity from signer extraction.

Users deposit ERC20 tokens to the InstructionSender contract (which acts as the vault) before trading. The TEE maintains per-user balances, validates orders against available funds, and signs withdrawal authorizations that users submit on-chain to claim their funds.

This is a reference demo -- prioritizing clarity, clean code, and good error handling over performance or security hardening.

## Architecture

**Approach: Separated Engine Package + Balance Layer**

The matching engine lives in `pkg/orderbook/` with a clean public API. The balance manager lives in `pkg/balance/` and tracks per-(user, token) balances. The extension layer in `internal/extension/` is a thin adapter that decodes TEE instructions, validates balances, calls the engine, updates balances, and builds results.

```
pkg/orderbook/        # Zero TEE/Flare dependencies. Independently testable.
pkg/balance/          # Balance tracking per (user, token). No TEE dependencies.
internal/extension/   # Thin adapter. Decodes instructions, validates, calls engine, returns results.
```

This separation means:
- Anyone can understand the matching engine without knowing TEE/Flare.
- Anyone can understand balance management without knowing orderbook internals.
- Anyone can understand the TEE integration without knowing either.
- Each file stays under 200 lines.

func NewBalanceManager() *BalanceManager

func (bm *BalanceManager) GetAll(user string) map[common.Address]TokenBalance

**Hold/Release pattern**: When a user places a limit order, the required funds are moved from `Available` to `Held`. When the order is matched, `Held` is debited and the counterparty is credited. When cancelled, `Held` is released back to `Available`.

- **Buy order hold**: `quantity * price` of the quote token (e.g., USDT for FLR/USDT)
- **Sell order hold**: `quantity` of the base token (e.g., FLR for FLR/USDT)

## Matching Engine

### Data Structures

Each `OrderBook` (one per trading pair) contains two `OrderSide` instances (bids and asks).

**OrderSide** uses:
- **Red-black tree** (`emirpasic/gods`): keyed by price, values are `*list.List` (FIFO order queues)
- **Order map** (`map[string]*orderLocation`): O(1) lookup for cancellations
- Bids tree: descending order (best bid = max)
- Asks tree: ascending order (best ask = min)

```go
type OrderSide struct {
    priceLevels *rbt.Tree
    orders      map[string]*orderLocation
    volume      uint64
    count       int
}

type orderLocation struct {
    price   uint64
    element *list.Element
}
```

### OrderBook Public API

```go
func NewOrderBook(pair string) *OrderBook
func (ob *OrderBook) PlaceLimitOrder(order *Order) ([]Match, error)
func (ob *OrderBook) PlaceMarketOrder(order *Order) ([]Match, error)
func (ob *OrderBook) CancelOrder(orderID string, owner string) (*Order, error)
func (ob *OrderBook) Depth() (bids []PriceLevel, asks []PriceLevel)
```

The matching engine itself is **not aware of balances**. Balance validation and hold/release happen in the extension layer before and after calling the engine. This keeps the engine pure and independently testable.

### Matching Rules

- **Price-time priority**: best price first, then FIFO within the same price level.
- **Execution price**: always the resting order's price (the order already on the book).
- **Multi-level sweep**: an incoming order walks through multiple price levels until fully filled or no more matching prices remain.
- **Partial fills**: supported. Unfilled remainder of a limit order rests on the book. Unfilled remainder of a market order is discarded.
- **Market order with no liquidity**: returns `ErrNoLiquidity`.
- **Buy matching condition**: incoming buy matches asks where `ask.price <= buy.price`.
- **Sell matching condition**: incoming sell matches bids where `bid.price >= sell.price`.

### Mutex Strategy

- Mutex at the `OrderBook` level (per pair). Different pairs don't block each other.
- `sync.RWMutex`: write lock for place/cancel, read lock for depth queries.

## Extension Layer

### Instruction Routing

Single OPType `ORDERBOOK` with six OPCommands:

| OPCommand | Handler | Description |
|-----------|---------|-------------|
| `DEPOSIT` | `processDeposit` | Credit user balance after on-chain ERC20 transfer |
| `WITHDRAW` | `processWithdraw` | Debit balance, return TEE-signed withdrawal params |
| `PLACE_ORDER` | `processPlaceOrder` | Validate balance, hold funds, place order |
| `CANCEL_ORDER` | `processCancelOrder` | Cancel order, release held funds |
| `GET_MY_STATE` | `processGetMyState` | Return user's balances, open orders, matches (encrypted) |
| `EXPORT_HISTORY` | `processExportHistory` | Return full history (encrypted), admin can target other users |

Routing follows the scaffold pattern:
```
processAction (OPType switch)
  +-- processOrderbook (OPCommand switch)
        +-- processDeposit
        +-- processWithdraw
        +-- processPlaceOrder
        +-- processCancelOrder
        +-- processGetMyState
        +-- processExportHistory
```

### Instruction Paths

**On-chain path** (deposits, withdrawals):
```
1. User calls InstructionSender contract function (deposit or withdraw)
2. For deposits: contract transfers ERC20 from user to itself (vault)
3. Contract packs TeeInstructionParams with claimBackAddress = msg.sender
4. Contract calls ITeeExtensionRegistry.sendInstructions()
5. Data providers detect instruction event, augment with signatures
6. Proxy receives augmented instruction, creates Action
7. TEE machine pulls Action from proxy queue
8. TEE forwards to extension via POST /action
9. Extension reads claimBackAddress as user identity
10. Handler validates and executes (credit balance / sign withdrawal params)
11. Returns ActionResult
12. TEE signs result, posts to proxy
13. User polls GET /action/result/<instructionId>
14. For withdrawals: user submits signed params to contract's executeWithdrawal()
```

**Direct instruction path** (orders, cancels, state, history):
```
1. User signs instruction payload off-chain (encrypted with TEE public key)
2. Submits to proxy via POST /direct-instruction
3. Proxy validates, creates Action with submissionTag="submit"
4. TEE machine pulls Action from proxy queue
5. TEE forwards to extension via POST /action
6. Extension extracts signer address from instruction signatures
7. Handler validates balances, calls engine, updates state
8. Returns ActionResult (matches, confirmations, or encrypted state)
9. TEE signs result, posts to proxy
10. User polls GET /action/result/<actionId>
```

### User Identity

Two instruction paths, two identity mechanisms:

| Path | Used for | Identity source |
|------|----------|----------------|
| On-chain instructions | DEPOSIT, WITHDRAW | `claimBackAddress` from `TeeInstructionParams` (set to `msg.sender` by the contract) |
| Direct instructions | PLACE_ORDER, CANCEL_ORDER, GET_MY_STATE, EXPORT_HISTORY | Signer address extracted from instruction signatures via `crypto.Ecrecover` |

Both mechanisms resolve to the same Ethereum address for a given user. This address:
- Becomes the `Order.Owner` field for order placement
- Is the identity key for balance lookups
- Authorizes cancellations (signer must match order owner)
- Determines whose state/history to return for GET_MY_STATE and EXPORT_HISTORY

### State Endpoint (Public)

`GET /state` returns the full orderbook depth for all configured pairs, plus recent match history. No instruction needed -- this is a direct HTTP query to the extension. This data is **public and unencrypted**.

## Vault & Deposits

### Vault Architecture

The InstructionSender contract itself acts as the vault -- it holds all deposited ERC20 tokens. Withdrawals are authorized by the TEE: the extension signs withdrawal parameters with the TEE node's key, and the user submits those signed parameters to the contract's `executeWithdrawal()` function, which verifies the TEE signature and transfers tokens.

```
+-- InstructionSender Contract (= Vault)
|   +-- Holds all deposited ERC20 tokens
|   +-- deposit(): transferFrom user, send DEPOSIT instruction to TEE
|   +-- withdraw(): send WITHDRAW instruction to TEE
|   +-- executeWithdrawal(): verify TEE signature, transfer tokens to user
|   +-- teeAddress: the TEE node's address (set once by admin after TEE starts)
```

No PMW keys needed. The TEE node's own key (the `teeId` address) is the authorized signer for withdrawals.

### InstructionSender Contract (Vault)

The contract serves dual purpose: it sends on-chain instructions to the TEE and holds deposited ERC20 tokens. Orders, cancels, state queries, and history export go through direct instructions (off-chain) and don't touch this contract.

```solidity
// contracts/InstructionSender.sol

contract InstructionSender {
    ITeeExtensionRegistry public registry;
    ITeeMachineRegistry public machineRegistry;
    uint256 public extensionId;

    // TEE node address -- authorized signer for withdrawals
    address public teeAddress;
    bool public teeAddressSet;

    // Admin allowlist for deposits (KYC gate)
    mapping(address => bool) public allowed;
    address[] public admins;

    // Replay protection for withdrawals
    mapping(bytes32 => bool) public usedWithdrawalIds;

    bytes32 public constant OP_TYPE_ORDERBOOK = bytes32("ORDERBOOK");
    bytes32 public constant OP_COMMAND_DEPOSIT = bytes32("DEPOSIT");
    bytes32 public constant OP_COMMAND_WITHDRAW = bytes32("WITHDRAW");

    constructor(
        address _registry,
        address _machineRegistry,
        address[] memory _admins
    ) {
        registry = ITeeExtensionRegistry(_registry);
        machineRegistry = ITeeMachineRegistry(_machineRegistry);
        admins = _admins;
        for (uint i = 0; i < _admins.length; i++) {
            allowed[_admins[i]] = true;  // admins are always allowed
        }
    }

    // --- Setup functions (one-time) ---

    /// @notice Set the TEE node address (authorized withdrawal signer). Called once after TEE starts.
    function setTeeAddress(address _teeAddress) external onlyAdmin {
        require(!teeAddressSet, "TEE address already set");
        require(_teeAddress != address(0), "zero address");
        teeAddress = _teeAddress;
        teeAddressSet = true;
    }

    // --- Admin functions ---

    modifier onlyAdmin() {
        bool isAdmin = false;
        for (uint i = 0; i < admins.length; i++) {
            if (admins[i] == msg.sender) { isAdmin = true; break; }
        }
        require(isAdmin, "not admin");
        _;
    }

    function allowUser(address user) external onlyAdmin {
        allowed[user] = true;
    }

    function removeUser(address user) external onlyAdmin {
        allowed[user] = false;
    }

    // --- Deposit (KYC-gated, on-chain instruction) ---

    function deposit(address token, uint256 amount) external payable {
        require(allowed[msg.sender], "not allowed to deposit");
        require(amount > 0, "zero amount");

        // Transfer ERC20 from user to this contract (vault)
        IERC20(token).transferFrom(msg.sender, address(this), amount);

        // Send DEPOSIT instruction to TEE
        bytes memory message = abi.encode(token, amount);
        _sendInstruction(OP_COMMAND_DEPOSIT, message);
    }

    // --- Withdraw (on-chain instruction to request, then executeWithdrawal to claim) ---

    /// @notice Request a withdrawal. TEE will return signed authorization params.
    function withdraw(bytes calldata message) external payable {
        _sendInstruction(OP_COMMAND_WITHDRAW, message);
    }

    /// @notice Execute a TEE-authorized withdrawal. Anyone can call with valid params.
    /// @param token ERC20 token address
    /// @param amount Amount to withdraw
    /// @param to Destination address
    /// @param withdrawalId Unique ID (e.g., instruction ID) for replay protection
    /// @param signature TEE node's signature over keccak256(token, amount, to, withdrawalId)
    function executeWithdrawal(
        address token,
        uint256 amount,
        address to,
        bytes32 withdrawalId,
        bytes calldata signature
    ) external {
        require(teeAddressSet, "TEE not configured");
        require(!usedWithdrawalIds[withdrawalId], "withdrawal already executed");

        // Verify TEE signature
        bytes32 hash = keccak256(abi.encodePacked(token, amount, to, withdrawalId));
        bytes32 ethHash = keccak256(abi.encodePacked("\x19Ethereum Signed Message:\n32", hash));
        (uint8 v, bytes32 r, bytes32 s) = splitSignature(signature);
        address signer = ecrecover(ethHash, v, r, s);
        require(signer != address(0) && signer == teeAddress, "invalid TEE signature");

        // Mark as used and transfer
        usedWithdrawalIds[withdrawalId] = true;
        IERC20(token).transfer(to, amount);
    }

    // --- Internal ---

    function _sendInstruction(bytes32 opCommand, bytes memory message) internal {
        address[] memory teeIds = machineRegistry.getRandomTeeIds(extensionId, 1);
        ITeeExtensionRegistry.TeeInstructionParams memory params = ITeeExtensionRegistry
            .TeeInstructionParams({
                opType: OP_TYPE_ORDERBOOK,
                opCommand: opCommand,
                message: message,
                cosigners: new address[](0),
                cosignersThreshold: 0,
                claimBackAddress: msg.sender
            });
        registry.sendInstructions{value: msg.value}(teeIds, params);
    }

    function setExtensionId() external {
        extensionId = registry.getExtensionIdByDeployer(address(this));
        require(extensionId != 0, "extension not registered");
    }
}
```

### Deposit Flow (Detail)

```
User                    Contract (Vault)           TEE Extension
 |                         |                            |
 |-- ERC20.approve(contract, amount) -->|               |
 |-- deposit(token, amount) -->|                        |
 |                         |-- transferFrom(user, self, amount)
 |                         |-- sendInstruction(DEPOSIT, ...) -->|
 |                         |                            |
 |                         |              decode instruction
 |                         |              read claimBackAddress as user
 |                         |              credit balance[user][token] += amount
 |                         |              record deposit in history
 |                         |              return DepositResponse
 |                         |                            |
 |<-- poll result: { token, amount, available } --------|
```

**Deposit validation in TEE**: The TEE trusts that the on-chain transfer happened because the instruction was sent by the InstructionSender contract (which does the `transferFrom` atomically with the instruction). The data providers' signatures on the instruction event provide consensus that this on-chain event occurred.

## Withdrawals

### Withdrawal Flow (Detail)

The TEE does **not** construct an Ethereum transaction. Instead, it signs the withdrawal parameters (token, amount, destination, withdrawal ID) with the TEE node's own key. The user takes these signed parameters and calls `executeWithdrawal()` on the contract, which verifies the signature and transfers tokens.

```
User                    Contract (Vault)       TEE Extension         TEE Sign Server
 |                         |                      |                       |
 |-- withdraw(message) --->|                      |                       |
 |                         |-- sendInstruction(WITHDRAW, ...) ->|         |
 |                         |                      |                       |
 |                         |          decode instruction                  |
 |                         |          read claimBackAddress as user        |
 |                         |          verify balance[user][token] >= amount|
 |                         |          debit balance[user][token] -= amount |
 |                         |          hash = keccak256(token, amount, to, withdrawalId)
 |                         |          POST /sign (TEE node key) -------->|
 |                         |                      |     sign hash         |
 |                         |                      |<--- SignResponse -----|
 |                         |          record withdrawal in history        |
 |                         |          return WithdrawResponse             |
 |                         |                      |                       |
 |<-- poll result: { token, amount, to, withdrawalId, signature } -------|
 |                         |                      |                       |
 |-- executeWithdrawal(token, amount, to, withdrawalId, signature) ->|   |
 |                         |-- verify TEE signature                       |
 |                         |-- check !usedWithdrawalIds[withdrawalId]     |
 |                         |-- IERC20.transfer(to, amount)                |
 |<-- tokens received -----|                                              |
```

### Signed Parameters (not a transaction)

The extension signs a parameter hash, not a raw Ethereum transaction. This eliminates the need for:
- Chain access from the extension (no nonce/gas lookups)
- PMW key management
- Transaction construction logic

The withdrawal ID uses the instruction ID from the on-chain instruction, providing natural replay protection -- each instruction ID can only be used once for withdrawal execution.

## User State & History

### GET_MY_STATE

Returns the calling user's current state: balances, open orders, and recent matches. The response is returned as ActionResult data. Since the instruction flow already encrypts responses with the user's key, no additional authentication is needed -- anyone can request someone's state, but only the owner can decrypt the response.

```go
func (e *Extension) processGetMyState(action teetypes.Action, df *instruction.DataFixed) teetypes.ActionResult {
    signer := extractSigner(action)

    e.mu.RLock()
    resp := types.GetMyStateResponse{
        Balances:   e.balances.GetAll(signer),
        OpenOrders: e.getUserOpenOrders(signer),
        Matches:    e.getUserMatches(signer),
    }
    e.mu.RUnlock()

    data, _ := json.Marshal(resp)
    return buildResult(action, df, data, 1, nil)
}
```

### EXPORT_HISTORY

Returns the full trading history for a user: all orders (open, filled, cancelled), all matches, all deposits, all withdrawals. Available off-chain via the proxy (poll the action result).

- **Self-export**: any user can export their own history
- **Admin export**: admin addresses can specify a `targetUser` to export another user's history

```go
func (e *Extension) processExportHistory(action teetypes.Action, df *instruction.DataFixed) teetypes.ActionResult {
    signer := extractSigner(action)

    var req types.ExportHistoryRequest
    // decode request (may be empty for self-export)

    targetUser := signer
    if req.TargetUser != "" {
        if !e.admins[signer] {
            return buildResult(action, df, nil, 0, fmt.Errorf("not authorized to export other user's history"))
        }
        targetUser = req.TargetUser
    }

    resp := e.history.Export(targetUser)
    data, _ := json.Marshal(resp)
    return buildResult(action, df, data, 1, nil)
}
```

## Admin Allowlist (KYC)

### Design

A simple allowlist that gates deposits only. Once a user has deposited, they can trade, cancel, and withdraw freely.

- **Admin list**: hardcoded at contract deploy time (constructor parameter). Not modifiable after deployment.
- **Allowlist management**: admins call `allowUser(address)` and `removeUser(address)` on the InstructionSender contract.
- **Enforcement**: the `deposit()` function checks `require(allowed[msg.sender])`. All other functions have no allowlist check.
- **Admins are always allowed**: admin addresses are added to the allowlist in the constructor.

### Scope

| Operation | Allowlist check? |
|-----------|-----------------|
| `deposit` | Yes -- must be on allowlist |
| `placeOrder` | No |
| `cancelOrder` | No |
| `withdraw` | No |
| `getMyState` | No |
| `exportHistory` | No (admin check is in TEE for target user export) |

### Market Order Balance Holds

For limit orders, the hold amount is deterministic: `quantity * price` (quote token) for buys, `quantity` (base token) for sells.

For market orders, the execution price is unknown upfront. The strategy is:
- **Hold the user's entire available balance** of the relevant token
- Execute the market order (sweeps price levels until filled or exhausted)
- **Release the unused portion** back to available after matching

This means a market buy holds all available quote token, and a market sell holds all available base token. Simple and safe -- the user can't overspend.

### TEE Address Setup

The contract needs the TEE node's address to verify withdrawal signatures. This address is the `teeId` -- derived from the TEE node's public key -- and is known after the TEE node starts and registers.

```
1. Deploy InstructionSender (admins set, no TEE address yet)
2. Start TEE node (teeId becomes known)
3. Register TEE version/machine
4. Admin calls setTeeAddress(teeId) on InstructionSender
5. Withdrawals are now enabled (deposits work immediately)
```

> **Future improvement: TEE address verification**
>
> The TEE node implements a `SignedKeyExistenceProof` mechanism -- a cryptographically signed
> attestation that a TEE machine holds a specific key. A hardened version of `setTeeAddress()`
> could verify on-chain that the provided address corresponds to a registered TEE machine for
> this extension, by checking against the `TeeMachineRegistry`. This would prevent an admin
> from setting an arbitrary address. For this demo, we trust the admin; on-chain verification
> is a natural next step for production.

## Design Constraints

- **In-memory only**: State is lost on restart. No persistence layer. A production version would use event sourcing from on-chain deposit/withdraw/order events to rebuild state on restart.
- **No order expiry**: Orders rest until matched or cancelled.
- **No order amendments**: Cancel and re-place instead.
- **uint64 arithmetic**: Prices and quantities are integer smallest-units. No decimal library.
- **Unbounded match/history**: The history slices grow without limit. Acceptable for a demo; a production version would cap or rotate.
- **Single TEE key for withdrawals**: One TEE node key authorizes all withdrawals. A production version might verify against TeeMachineRegistry to support TEE rotation.
- **Two-step withdrawal**: The TEE signs withdrawal parameters but does not submit anything on-chain. The user calls `executeWithdrawal()` on the contract with the signed params.
- **KYC gate is on-chain only**: The allowlist is enforced by the InstructionSender contract's `deposit()` function. The TEE itself does not re-check the allowlist (once the instruction arrives, the deposit was valid on-chain).
- **Trading pairs are token-agnostic in the engine**: The matching engine uses string pair names (e.g., "FLR/USDT"). The extension layer maps pair names to (base token address, quote token address) for balance operations.
- **Market orders hold full balance**: A market buy holds the user's entire available quote token balance; unused portion is released after matching. This is simple but means a user can only have one market order at a time.
- **Dual instruction paths**: Deposits/withdrawals use on-chain instructions (identity from claimBackAddress). Orders/cancels/state use direct instructions (identity from signer extraction). Both resolve to the same Ethereum address.
