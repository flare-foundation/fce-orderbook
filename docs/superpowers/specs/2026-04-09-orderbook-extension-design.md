# Orderbook Extension Design Spec

## Overview

A clean, well-documented orderbook extension for Flare Confidential Compute (FCC), built on the extension-scaffold. Implements a multi-pair limit/market order matching engine that receives orders via direct instructions and executes price-time priority matching in a TEE.

This is a reference demo — prioritizing clarity, clean code, and good error handling over performance or security hardening.

## Architecture

**Approach: Separated Engine Package**

The matching engine lives in `pkg/orderbook/` with a clean public API. The extension layer in `internal/extension/` is a thin adapter that decodes TEE instructions, calls the engine, and builds results.

```
pkg/orderbook/        # Zero TEE/Flare dependencies. Independently testable.
internal/extension/   # Thin adapter. Decodes instructions, calls engine, returns results.
```

This separation means:
- Anyone can understand the matching engine without knowing TEE/Flare.
- Anyone can understand the TEE integration without knowing orderbook internals.
- Each file stays under 200 lines.

## Core Types

### Orderbook Domain Types (`pkg/orderbook/order.go`)

```go
type Side string
const (
    Buy  Side = "buy"
    Sell Side = "sell"
)

type OrderType string
const (
    Limit  OrderType = "limit"
    Market OrderType = "market"
)

type Order struct {
    ID        string
    Owner     string     // signer address for cancellation auth
    Pair      string
    Side      Side
    Type      OrderType
    Price     uint64     // smallest units, 0 for market orders
    Quantity  uint64     // original quantity
    Remaining uint64     // unfilled quantity
    Timestamp int64      // from instruction timestamp
}

type Match struct {
    BuyOrderID  string
    SellOrderID string
    Pair        string
    Price       uint64   // execution price (resting order's price)
    Quantity    uint64
    Timestamp   int64
}

type PriceLevel struct {
    Price      uint64
    Quantity   uint64   // aggregated remaining across all orders at this price
    OrderCount int
}
```

### Request/Response Types (`pkg/types/types.go`)

```go
// PlaceOrderRequest is sent as the originalMessage in a PLACE_ORDER instruction.
type PlaceOrderRequest struct {
    Pair     string           `json:"pair"`
    Side     orderbook.Side   `json:"side"`
    Type     orderbook.OrderType `json:"type"`
    Price    uint64           `json:"price"`
    Quantity uint64           `json:"quantity"`
}

// PlaceOrderResponse is returned in ActionResult.Data.
type PlaceOrderResponse struct {
    OrderID   string            `json:"orderId"`
    Status    string            `json:"status"`    // "filled", "partial", "resting"
    Matches   []orderbook.Match `json:"matches,omitempty"`
    Remaining uint64            `json:"remaining"`
}

// CancelOrderRequest is sent as the originalMessage in a CANCEL_ORDER instruction.
type CancelOrderRequest struct {
    OrderID string `json:"orderId"`
}

// CancelOrderResponse is returned in ActionResult.Data.
type CancelOrderResponse struct {
    OrderID   string `json:"orderId"`
    Pair      string `json:"pair"`
    Side      string `json:"side"`
    Remaining uint64 `json:"remaining"`
}

// State is returned by GET /state.
type State struct {
    Pairs      map[string]PairState `json:"pairs"`
    MatchCount int                  `json:"matchCount"`
    Matches    []orderbook.Match    `json:"matches"`
}

type PairState struct {
    Bids []orderbook.PriceLevel `json:"bids"`
    Asks []orderbook.PriceLevel `json:"asks"`
}

type StateResponse struct {
    StateVersion common.Hash `json:"stateVersion"`
    State        State       `json:"state"`
}
```

## Matching Engine

### Data Structures

Each `OrderBook` (one per trading pair) contains two `OrderSide` instances (bids and asks).

**OrderSide** uses:
- **Red-black tree** (`emirpasic/gods`): keyed by price, values are `*list.List` (FIFO order queues)
- **Order map** (`map[string]*orderLocation`): O(1) lookup for cancellations
- Bids tree: descending order (best bid = max)
- Asks tree: ascending order (best ask = min)

```go
type OrderSide struct {
    priceLevels *rbt.Tree
    orders      map[string]*orderLocation
    volume      uint64
    count       int
}

type orderLocation struct {
    price   uint64
    element *list.Element
}
```

### OrderBook Public API

```go
func NewOrderBook(pair string) *OrderBook
func (ob *OrderBook) PlaceLimitOrder(order *Order) ([]Match, error)
func (ob *OrderBook) PlaceMarketOrder(order *Order) ([]Match, error)
func (ob *OrderBook) CancelOrder(orderID string, owner string) (*Order, error)
func (ob *OrderBook) Depth() (bids []PriceLevel, asks []PriceLevel)
```

### Matching Rules

- **Price-time priority**: best price first, then FIFO within the same price level.
- **Execution price**: always the resting order's price (the order already on the book).
- **Multi-level sweep**: an incoming order walks through multiple price levels until fully filled or no more matching prices remain.
- **Partial fills**: supported. Unfilled remainder of a limit order rests on the book. Unfilled remainder of a market order is discarded.
- **Market order with no liquidity**: returns `ErrNoLiquidity`.
- **Buy matching condition**: incoming buy matches asks where `ask.price <= buy.price`.
- **Sell matching condition**: incoming sell matches bids where `bid.price >= sell.price`.

### Mutex Strategy

- Mutex at the `OrderBook` level (per pair). Different pairs don't block each other.
- `sync.RWMutex`: write lock for place/cancel, read lock for depth queries.

## Extension Layer

### Instruction Routing

Single OPType `ORDERBOOK` with two OPCommands:

| OPCommand | Handler | Description |
|-----------|---------|-------------|
| `PLACE_ORDER` | `processPlaceOrder` | Place a limit or market order |
| `CANCEL_ORDER` | `processCancelOrder` | Cancel a resting order by ID |

Routing follows the scaffold pattern:
```
processAction (OPType switch)
  └── processOrderbook (OPCommand switch)
        ├── processPlaceOrder
        └── processCancelOrder
```

### Extension Struct

```go
type Extension struct {
    mu         sync.RWMutex
    Server     *http.Server
    orderbooks map[string]*orderbook.OrderBook  // pair -> orderbook
    matches    []orderbook.Match                // all matches across pairs
    orders     map[string]string                // orderID -> pair (for cancel routing)
}
```

### Instruction Path (Direct Instructions)

```
1. User signs order payload off-chain
2. Submits to proxy via POST /direct-instruction
3. Proxy validates signatures, creates Action with submissionTag="submit"
4. TEE machine pulls Action from proxy queue
5. TEE forwards to extension via POST /action
6. Extension decodes, routes to handler
7. Handler extracts signer, calls orderbook engine
8. Returns ActionResult (matches or confirmation)
9. TEE signs result, posts to proxy
10. User polls GET /action/result/<actionId>
```

### Signer-Based Ownership

The signer's Ethereum address is extracted from the direct instruction signatures using `crypto.Ecrecover`. This address becomes the `Order.Owner` field. Cancellations verify that the cancel instruction's signer matches the order's owner.

### State Endpoint

`GET /state` returns the full orderbook depth for all configured pairs, plus recent match history. No instruction needed — this is a direct HTTP query to the extension.

## Error Handling

Three layers with clear responsibilities:

### Layer 1: Orderbook Engine (`pkg/orderbook/`)

Sentinel errors for domain problems. No logging — caller decides.

```go
var (
    ErrOrderNotFound   = errors.New("order not found")
    ErrNotOwner        = errors.New("not the order owner")
    ErrNoLiquidity     = errors.New("no liquidity available")
    ErrInvalidPrice    = errors.New("price must be greater than zero")
    ErrInvalidQuantity = errors.New("quantity must be greater than zero")
    ErrInvalidSide     = errors.New("invalid order side")
    ErrInvalidType     = errors.New("invalid order type")
)
```

### Layer 2: Extension Handlers (`internal/extension/`)

Wraps engine errors with context, maps to ActionResult status codes, logs via Flare logger.

```go
// error → ActionResult with Status=0, Log="error: placing order: invalid order side"
return buildResult(action, df, nil, 0, fmt.Errorf("placing order: %w", err))
```

### Layer 3: HTTP Transport (`internal/extension/utils.go`)

- Malformed HTTP body → 400
- Unknown OPType/OPCommand → 501
- JSON encoding failure → 500

## Decoder Registration

All request and response types registered with the decoder registry for the types-server:

```go
func RegisterDecoders(r *decoder.Registry) {
    // PLACE_ORDER message + result
    r.Register(Key("ORDERBOOK", "PLACE_ORDER", KindMessage), NewJSONDecoder[PlaceOrderRequest]())
    r.Register(Key("ORDERBOOK", "PLACE_ORDER", KindResult),  NewJSONDecoder[PlaceOrderResponse]())
    // CANCEL_ORDER message + result
    r.Register(Key("ORDERBOOK", "CANCEL_ORDER", KindMessage), NewJSONDecoder[CancelOrderRequest]())
    r.Register(Key("ORDERBOOK", "CANCEL_ORDER", KindResult),  NewJSONDecoder[CancelOrderResponse]())
}
```

No TEE dependencies. Tests the matching engine in isolation.

|------|----------|

| `TestPlaceLimitSell_NoMatch` | Sell rests when no bids exist |

| `TestLimitOrder_PartialFill` | Remainder rests on book |
| `TestLimitOrder_MultiLevelSweep` | Incoming sweeps multiple price levels |
| `TestMarketOrder_FullFill` | Market fills against best prices |
| `TestMarketOrder_NoLiquidity` | Returns ErrNoLiquidity |
| `TestMarketOrder_PartialFill` | Fills available, discards remainder |

| `TestCancelOrder_NotOwner` | Returns ErrNotOwner |

## Design Constraints

- **In-memory only**: State is lost on restart. No persistence layer.
- **No smart contract instruction path**: Orders submitted via direct instructions only. InstructionSender.sol kept for future use.
- **No order expiry**: Orders rest until matched or cancelled.
- **No order amendments**: Cancel and re-place instead.
- **uint64 arithmetic**: Prices and quantities are integer smallest-units. No decimal library.
- **Unbounded match history**: The `matches` slice in the Extension struct grows without limit. Acceptable for a demo; a production version would cap or rotate.
